# Verteilung

Die mobile 027Apps-App verwendet Expo Application Services (EAS) zum Erstellen, Aktualisieren und Verteilen der Anwendung.

---

## Dual-App-Varianten

Die App verwendet `app.config.ts` (dynamische Konfiguration), um je nach Umgebungsvariable `EXPO_PUBLIC_APP_VARIANT` zwei verschiedene Anwendungen zu erzeugen:

| | Beta | Produktion |
|---|---|---|
| App-Name | 027Apps Beta | 027Apps |
| Paket | `com.rferic.apps027.beta` | `com.rferic.apps027` |
| Schema | `027apps-beta://` | `027apps://` |
| Standard-API-URL | Vercel Preview-URL | `https://027apps.com` |
| OTA-Kanal | `beta` | `production` |
| Modulsichtbarkeit | Alle (inkl. Beta-Module) | Nur stabile Module |

### Wie es funktioniert

```ts
// app.config.ts (vereinfacht)
const VARIANT = process.env.EXPO_PUBLIC_APP_VARIANT ?? 'production'
const isBeta = VARIANT === 'beta'

export default { expo: {
  name: isBeta ? '027Apps Beta' : '027Apps',
  ios: { bundleIdentifier: isBeta ? 'com.rferic.apps027.beta' : 'com.rferic.apps027' },
  android: { package: isBeta ? 'com.rferic.apps027.beta' : 'com.rferic.apps027' },
  extra: {
    appVariant: isBeta ? 'beta' : 'production',
    defaultApiUrl: isBeta ? 'https://027apps.vercel.app' : 'https://027apps.com',
  },
}}
```

Beide Varianten kΟΕnnen gleichzeitig auf demselben GerΟΛt installiert werden.

## Versionsverwaltung

Die App-Version wird in `app.config.ts` unter `expo.version` definiert.

### VersionsbeschrΟΛnkungen

Jede Variante hat ihre eigene Versionsschiene auf dem Server:

| Variante | Server-SchlΟΦssel | Endpunkt |
|---|---|---|
| Produktion | `app_settings.mobile_version` | `GET /api/v1/mobile/version` |
| Beta | `app_settings.mobile_version_beta` | `GET /api/v1/mobile/version?variant=beta` |

Beim Start ruft die App den variantenabhΟΛngigen Endpunkt auf. Wenn `current_version < min_version`, blockiert `UpdateRequiredScreen` die UI.

### Eine Version verΟΕffentlichen (ΟΦber CI)

1. Gehe zu GitHub Actions βÜ£ **Mobile Build (APK)**
2. WΟΛhle das Profil: `beta` oder `production`
3. Gib den Versions-Tag ein (z. B. `1.0.1`)
4. FΟΦhre den Workflow aus

Der CI-Workflow automatisch:
1. Erstellt die APK ΟΦber EAS Build
2. LΟΛdt das Artefakt herunter
3. LΟΛdt es in Supabase Storage hoch (`apk-releases/` fΟΦr Produktion, `apk-releases/beta/` fΟΦr Beta)

## EAS Update (OTA)

Over-the-Air-Updates senden JavaScript-/Asset-Ο³nderungen ohne einen neuen nativen Build.

### KanΟΛle

Jede Variante abonniert ihren eigenen OTA-Kanal (definiert in `eas.json` ΟΦber die Eigenschaft `channel`):

- `production`-Kanal βÜ£ Produktions-Builds erhalten OTA-Updates.
- `beta`-Kanal βÜ£ Beta-Builds erhalten OTA-Updates.

### VerΟΕffentlichung

Updates werden ΟΦber `mobile-ota.yml` beim Push auf `main` mit Ο³nderungen in `mobile/` verΟΕffentlicht. Der Workflow verΟΕffentlicht auf **beiden** KanΟΛlen und stellt sicher, dass beide Varianten die gleichen JS-Updates erhalten:

```yaml

strategy:
  matrix:
    channel: [production, beta]
steps:
  - run: npx eas update --channel ${{ matrix.channel }}
```

### Manuelles OTA

```bash
cd mobile
npx eas update --channel beta --message "Fix: Anmeldebildschirm"
npx eas update --channel production --message "Fix: Anmeldebildschirm"
```
