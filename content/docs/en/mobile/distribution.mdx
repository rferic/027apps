# Distribution

The 027Apps mobile app uses Expo Application Services (EAS) for building, updating, and distributing the application.

---

## Dual-App Variants

The app uses `app.config.ts` (dynamic config) to produce two distinct applications depending on the `EXPO_PUBLIC_APP_VARIANT` environment variable:

| | Beta | Production |
|---|---|---|
| App name | 027Apps Beta | 027Apps |
| Package | `com.rferic.apps027.beta` | `com.rferic.apps027` |
| Scheme | `027apps-beta://` | `027apps://` |
| Default API URL | Preview Vercel URL | `https://027apps.com` |
| OTA channel | `beta` | `production` |
| Module visibility | All (incl. beta modules) | Only stable modules |

### How it works

```ts
// app.config.ts (simplified)
const VARIANT = process.env.EXPO_PUBLIC_APP_VARIANT ?? 'production'
const isBeta = VARIANT === 'beta'

export default { expo: {
  name: isBeta ? '027Apps Beta' : '027Apps',
  ios: { bundleIdentifier: isBeta ? 'com.rferic.apps027.beta' : 'com.rferic.apps027' },
  android: { package: isBeta ? 'com.rferic.apps027.beta' : 'com.rferic.apps027' },
  extra: {
    appVariant: isBeta ? 'beta' : 'production',
    defaultApiUrl: isBeta ? 'https://027apps.vercel.app' : 'https://027apps.com',
  },
}}
```

Both variants can be installed on the same device simultaneously.

## Build Profiles

Four build profiles are defined in `eas.json`:

| Profile | Variant | Dev Client | Distribution | Channel |
|---|---|---|---|---|
| `development` | beta | Yes | Internal | — |
| `beta` | beta | No | Internal | `beta` |
| `preview` | beta | No | Internal | `beta` |
| `production` | production | No | — | `production` |

### eas.json

```json
{
  "build": {
    "beta": {
      "channel": "beta",
      "distribution": "internal",
      "android": { "buildType": "apk" },
      "env": {
        "EXPO_PUBLIC_APP_VARIANT": "beta",
        "EXPO_PUBLIC_BETA_API_URL": "https://027apps.vercel.app",
        "EXPO_PUBLIC_EAS_PROJECT_ID_BETA": "${EXPO_PUBLIC_EAS_PROJECT_ID_BETA}",
        "EXPO_PUBLIC_EAS_PROJECT_ID": "${EXPO_PUBLIC_EAS_PROJECT_ID}"
      }
    },
    "production": {
      "channel": "production",
      "android": { "buildType": "apk" },
      "env": {
        "EXPO_PUBLIC_APP_VARIANT": "production",
        "EXPO_PUBLIC_EAS_PROJECT_ID": "${EXPO_PUBLIC_EAS_PROJECT_ID}"
      }
    }
  }
}
```

## Version Management

The app version is defined in `app.config.ts` under `expo.version`.

### Version Constraints

Each variant has its own version track on the server:

| Variant | Server key | Endpoint |
|---|---|---|
| Production | `app_settings.mobile_version` | `GET /api/v1/mobile/version` |
| Beta | `app_settings.mobile_version_beta` | `GET /api/v1/mobile/version?variant=beta` |

On startup, the app calls the variant-aware endpoint. If `current_version < min_version`, `UpdateRequiredScreen` blocks the UI.

### Publishing a Version (via CI)

1. Go to GitHub Actions → **Mobile Build (APK)**
2. Select profile: `beta` or `production`
3. Enter version tag (e.g. `1.0.1`)
4. Run workflow

The CI workflow automatically:
1. Builds the APK via EAS Build
2. Downloads the artifact
3. Uploads to Supabase Storage (`apk-releases/` for production, `apk-releases/beta/` for beta)
4. Updates `app_settings` with the new version record

No manual uploads required.

## EAS Update (OTA)

Over-the-Air updates push JavaScript/asset changes without a new native build.

### Channels

Each variant subscribes to its own OTA channel (defined in `eas.json` via the `channel` property):

- `production` channel → production builds receive OTA updates.
- `beta` channel → beta builds receive OTA updates.

### Publishing

Updates are published via `mobile-ota.yml` on push to `main` with changes in `mobile/`. The workflow publishes to **both** channels, ensuring both variants receive the same JS updates:

```yaml
# mobile-ota.yml (simplified)
strategy:
  matrix:
    channel: [production, beta]
steps:
  - run: npx eas update --channel ${{ matrix.channel }}
```

### Manual OTA

```bash
cd mobile
npx eas update --channel beta --message "Fix: login screen"
npx eas update --channel production --message "Fix: login screen"
```

## CI/CD Pipelines

| Workflow | Trigger | Actions |
|---|---|---|
| `mobile-ci.yml` | PR affecting `mobile/` or `packages/contracts/` | Type check + tests |
| `mobile-ota.yml` | Push to `main` with `mobile/` changes | Type check + OTA to both channels |
| `mobile-build.yml` | Manual `workflow_dispatch` | EAS build + auto-upload to storage + version update |

### Required Secrets

- `EXPO_TOKEN` — Expo personal access token
- `SUPABASE_SERVICE_ROLE_KEY` — For storage uploads
- `NEXT_PUBLIC_SUPABASE_URL` — Supabase project URL
- `API_BASE_URL` — For the admin upload endpoint

---

## Auto-Cleanup

Push tokens are automatically cleaned up:
- Orphaned tokens (user deleted) are removed.
- Tokens not updated in 90 days are removed.

## Environment Variables

| Variable | Purpose | Where |
|---|---|---|
| `EXPO_PUBLIC_APP_VARIANT` | Determines beta vs production build | `app.config.ts` |
| `EXPO_PUBLIC_BETA_API_URL` | Default API URL for beta builds | `app.config.ts` |
| `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL | `supabase.ts` |
| `EXPO_PUBLIC_SUPABASE_ANON_KEY` | Supabase anonymous key | `supabase.ts` |
| `EXPO_PUBLIC_EAS_PROJECT_ID` | Production EAS project | `app.config.ts` |
| `EXPO_PUBLIC_EAS_PROJECT_ID_BETA` | Beta EAS project | `app.config.ts` |

## Self-Hosting

Users configure the server URL in the Welcome/onboarding screen or Settings. The app ships with the production default URL. Self-hosted instances are fully supported — just enter the custom URL during onboarding.
